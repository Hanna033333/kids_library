# 책자리 – 작업계획서 (for Antigravity)

## 0. 프로젝트 한 줄 요약
책자리는 **어떤 책을 읽혀야 할지 모르는 부모를 위한, AI 사서의 상황별 정서 맞춤 큐레이션 서비스**다.
발견한 책은 내 주변 도서관 소장/대출 여부 확인 및 교보문고 구매까지 바로 연결된다.

> **[피봇 기록 - 2026.05]** 서비스 핵심 가치를 "청구기호 기반 판교 로컬 유틸리티"에서 "전국 학부모 대상 큐레이션 플랫폼"으로 전환. 청구기호/도서관 연결 기능은 큐레이션 이후 탐색의 최종 Action 단계로 재정의.

---

## 1. 지금 단계의 목표
- **전국 단위 DAU 100명** 달성을 위한 큐레이션 피봇 안정화
- 홈 화면 큐레이션 → 도서 상세 → (도서관 대출 or 교보 구매) 흐름 완성
- 상황별 큐레이션 콘텐츠 확장 및 바이럴 공유 루프 구축

성공 기준:
- **[NSM] 도서 상세 페이지 DAU** (일별 유니크 도달자 수) — 홈/리스트/네이버 검색 등 모든 유입 경로 포괄
  > 네이버 검색을 통한 직접 착지가 주요 유입이므로, 최종 도달 페이지 기준으로 가치 창출 측정
- **[보조: 액션 전환] 행동 전환 수**: 교보 구매 클릭 + 대출 확인 클릭 합산
- **[보조: 인게이지먼트] 회원 가입 수 & 찜하기 수**: 비로그인 북마크 툴팁 등 마케팅 UX 최적화 성과 측정 및 진성 유저 확보 지표
- **[리텐션] 7일 재방문율 35% 이상** (큐레이션 정기 탐색 주기 기반 재방문)
- 청구기호/대출 정보는 상세 페이지 내 부가 정보로 자연스럽게 제공
- 모바일 기준 완전 사용 가능

## 2. 핵심 가치 제안 (Value Proposition)

### 메인 소구점: "큐레이션 First"
> **"어떤 책을 읽혀야 할지 모르는 부모를 위한 AI 사서의 상황별 정서 맞춤 큐레이션"**

- **Curation First**: 아이의 연령뿐 아니라 정서적 상태(잠자리, 감정 표현, 사회성 등)에 맞춘 고품질 큐레이션을 첫 화면에 제공하여 부모의 책 선택 고민을 해결
- **The Ultimate Conversion (도서관 연결)**: 차별점은 **"발견한 책을 바로 내 주변 도서관에서 대출할 수 있는지 확인"** 해주는 실용적 연결성. 교보문고 구매 링크와 도서관 소장 여부를 탐색의 최종 Action으로 묶어 독보적 입지 구축

### 기존 소구점 위상 재정의 (청구기호/대출 조회)
- 청구기호·대출 가능 여부 조회는 메인 가치가 아닌, **큐레이션 탐색 이후 "찾기 단계(Action)"의 보조 기능**으로 위상 재조정
- 로컬(판교) 유저에게는 여전히 유효한 기능이나, 전국 유저에게는 "발견" 경험을 우선 제공

## 3. 현재 포함 범위 (Scope)

### 포함
- **홈 화면 큐레이션 (최우선)**
  - 상황별·정서별 큐레이션 섹션 (잠자리, 감정 표현, 사회성, 계절 등)
  - 연령별 추천 큐레이션
  - 어린이도서연구회 추천 / 칼데콧 수상작 등 테마별 큐레이션
  - 메인 배너 (시즌별 프로모션)
  - 큐레이션 섹션 공유 기능 (카카오/SNS 바이럴 루프)

- **검색 및 필터링**
  - 제목/저자 기반 검색
  - 카테고리 필터 (그림책, 동화, 지식책 등)
  - 연령 필터 (0-2세, 3-4세, 5-7세, 8~13세)
  - 정렬 (최신순, 제목순)

- **책 리스트**
  - 무한 스크롤 (24권씩 로딩)
  - 책 표지 이미지 (고화질 렌더링)

- **책 상세 정보**
  - 기본 정보 (제목, 저자, 출판사, ISBN)
  - 청구기호 + 권 정보 (보조 정보로 표시)
  - 실시간 대출 가능 여부 (보조 정보로 표시)
  - 큐레이션 태그

- **도서 구매 연결 (Action)**
  - 교보문고 구매 링크 (LinkPrice) — 메인 CTA
  - 도서관 소장 여부 및 대출 가능 여부 — 보조 CTA

- **비로그인 북마크 (찜)**
  - 로컬스토리지 기반 즉시 찜 반영
  - 로그인 유도는 강제 팝업이 아닌 혜택 안내 툴팁으로 처리

- **오류 신고** (구글 폼)
- **회원 시스템** (소셜 로그인: 카카오, 구글)

### 제외
- 결제
- 도서 배송

## 4. 핵심 사용자 흐름

### Primary Flow (전국 유저)
1. 보호자가 상황에 맞는 책을 찾는다 ("잠들기 싫어하는 아이에게 줄 책")
2. 홈 큐레이션 섹션에서 책 카드를 발견한다
3. 책 상세 페이지로 바로 이동한다
4. 교보문고에서 구매하거나, 도서관 소장 여부를 확인한다

### Secondary Flow (판교/로컬 유저)
1. 보호자가 도서관 방문 전 책을 미리 파악한다
2. 큐레이션 or 검색으로 원하는 책을 찾는다
3. 청구기호를 보고 도서관에서 직접 찾는다
4. 정보가 틀리면 신고한다

## 5. 운영 원칙
- **큐레이션 콘텐츠 품질이 최우선**: 상황별·정서별 큐레이션의 신뢰도와 감성 코피가 재방문을 결정한다
- 데이터는 완벽하지 않아도 공개한다 — 대신 **고칠 수 있는 구조**를 먼저 만든다
- 기능 추가보다 "헷갈리지 않음"을 우선한다
- **"발견 → 행동(구매/대출)"의 흐름을 최단 경로로 유지**한다

## 7. API 설계

### 7-1. Base URL
- **Production**: `https://api.checkjari.com`
- **Development**: `http://127.0.0.1:8000`
- **API Prefix**: `/api`

> 환경변수 `NEXT_PUBLIC_API_URL`로 설정

### 7-2. 책 관련 API (`/api/books`)

#### GET `/api/books/search`
책 검색 및 필터링

**Query Parameters:**
- `q` (optional): 검색어 (제목, 저자, 키워드)
- `age` (optional): 연령대 (예: '0-2세', '3-4세', '5-7세', '초등 저학년', '초등 고학년')
- `category` (optional): 카테고리 (예: '그림책', '동화', '지식책')
- `sort` (optional): 정렬 기준 ('title' 또는 'pangyo_callno', 기본값: 'pangyo_callno')
- `page` (optional): 페이지 번호 (기본값: 1)
- `limit` (optional): 페이지당 항목 수 (기본값: 20, 최대: 100)

**Response:**
```json
{
  "data": [...],
  "total": 1234,
  "page": 1,
  "limit": 20
}
```

#### GET `/api/books/list`
전체 목록 조회 (pangyo_callno가 있는 책만)

**Query Parameters:**
- `age`, `category`, `sort`, `page`, `limit` (검색 API와 동일)

#### GET `/api/books/{book_id}`
책 상세 정보 조회

**Response:**
- 책 기본 정보
- 청구기호 및 권 정보
- 알라딘 API 도서 소개 (캐싱)
- 찜 횟수

#### POST `/api/books/loan-status`
여러 책의 대출 정보를 병렬로 조회

**Request Body:**
```json
{
  "book_ids": [1, 2, 3, 4, 5]
}
```

**Response:**
```json
{
  "1": { "status": "대출가능", "location": "어린이자료실" },
  "2": { "status": "대출중", "returnDate": "2026-01-15" }
}
```

#### POST `/api/books/by-ids`
여러 책의 상세 정보를 ID로 조회

**Request Body:**
```json
{
  "book_ids": [1, 2, 3]
}
```

### 7-3. 추천·큐레이션 API

#### GET `/api/recommendations/age/{age_group}`
연령별 추천 도서 목록

#### GET `/api/recommendations/curation/{tag}`
큐레이션 태그별 추천 도서 목록
- 홈 화면 노출 상위 7권은 전체 목록 페이지에서도 최상단 고정
- 칼데콧 등 특별 테마는 기본 정렬을 제목 오름차순(ㄱㄴㄷ)으로 적용

### 7-4. API 성능 최적화
- **캐싱**: 대출 상태 API는 5분 캐시
- **병렬 처리**: 여러 책의 대출 정보를 비동기로 병렬 조회
- **타임아웃**: 외부 API 호출 시 3-10초 타임아웃 설정
- **에러 핸들링**: 외부 API 실패 시 graceful degradation

## 8. 데이터 시스템 구조 (Data System Architecture)

### 8-1. 데이터 흐름 개요
책자리의 데이터는 **"신뢰 가능한 큐레이션 목록 → 도서관 소장/청구기호 매칭 → 사용자 검증(신고)"** 구조로 운영된다.

### 8-2. 데이터 소스
#### Primary Source
- **큐레이션 콘텐츠 (상황별·정서별·테마별)**
  - 어린이도서연구회 추천, 칼데콧 수상작, 편집팀 큐레이션
  - 서비스의 "발견 가치" 핵심 — 콘텐츠 품질이 리텐션을 결정

#### Secondary Source
- **Data4Library API**: 소장 여부 확인
- **도서관 장서 데이터 (판교 도서관)**: 웹 스크래핑을 통한 청구기호 수집
- **판교 도서관 대출 상태 API**: 실시간 대출 가능 여부 확인, 캐시 TTL 5분

---

### 8-3. 주요 테이블 구조 (Supabase)
#### 1) childbook_items
**기본 정보**
- id (Primary Key)
- title (제목)
- author (저자)
- publisher (출판사)
- isbn (ISBN)
- category (카테고리: 그림책, 동화, 지식책 등)
- age (연령: 0-2세, 3-4세, 5-7세, 초등 저학년, 초등 고학년)
- keywords (검색용 키워드)

**청구기호 관련**
- callno (청구기호 - 최종 사용)
- web_scraped_callno (웹 스크래핑으로 수집한 청구기호)
- vol (권 정보 - 시리즈물의 경우)

**큐레이션 및 메타데이터**
- curation_tag (큐레이션 태그)
- situation_tag (상황별 태그 - 신규: 잠자리, 감정표현, 사회성 등)
- similarity (유사도 점수)
- owned (도서관 소장 여부)

**타임스탬프**
- created_at
- updated_at

**역할:**
- 큐레이션 도서 목록 관리 (핵심 콘텐츠)
- 실제 도서관에서 찾기 위한 보조 정보 제공
- 검색 및 필터링 기준 데이터

---

### 8-4. 청구기호 매칭 원칙
- ISBN 기준 매칭을 1차로 사용
- 제목 기반 매칭은 보조 수단으로만 활용
- 자동 매칭 결과는 **완전 신뢰하지 않는다**
- 사용자 신고를 통한 보정 구조를 필수로 둔다
- 소장 정보 미존재 시 에러가 아닌 "준비 중" 플레이스홀더로 처리하고 교보 구매 CTA 활성화

---

### 8-5. 데이터 보정 & 운영 플로우

**청구기호 오류 신고 프로세스:**
1. 사용자가 책 상세 페이지에서 오류 발견
2. 구글 폼을 통해 신고 제출 (올바른 청구기호 텍스트 입력)
3. 운영자가 신고 내용 검토
4. Supabase에서 `childbook_items` 테이블 수동 수정
5. 수정 완료 (자동 반영)

---

### 8-6. 데이터 운영 원칙
- 큐레이션 콘텐츠 **품질과 감성**이 최우선 (재방문 결정 요인)
- 도서관 데이터 정확도보다 **접근성**을 우선
- 불완전한 데이터는 숨기지 않고 상태로 노출
- "틀릴 수 있다"는 전제를 구조로 해결
- 자동화보다 **검증 가능한 흐름** 유지

---

### 8-7. 향후 확장 가능성 (Future)
- 다중 도서관 지원 (library_code 분리) — 전국 단위 대출 조회
- AI 기반 상황별 큐레이션 자동화 (LLM 큐레이션 생성)
- 큐레이션 공유 카드 고도화 (카카오/SNS 바이럴 최적화)
- 책 추가 (큐레이션 확대, 영어 책 추가 등)
- 찜한 도서 공유 (북마크 → 소셜 공유)
- 홈 추천 섹션 확대 (주제별, 계절별, 테마별)

## 10. 배포 전략 (Deployment Strategy)

### 10-1. 브랜치 전략
- **dev 브랜치**: 개발 및 테스트용 배포
- **main 브랜치**: 프로덕션 배포

### 10-2. 배포 프로세스

#### Frontend (Vercel)
1. **개발 환경 배포**
   - `dev` 브랜치에 푸시
   - Vercel이 자동으로 preview 환경에 배포
   - 테스트 및 검증 수행

2. **프로덕션 배포**
   - 여러 기능을 `dev` 브랜치에서 충분히 테스트
   - 검증 완료 후 승인 요청
   - 승인 받은 후 `main` 브랜치에 일괄 병합
   - Vercel이 자동으로 프로덕션 환경에 배포

#### Backend (Render)
- `main` 브랜치 푸시 시 자동 배포
- 배포 전 로컬 테스트 필수

### 10-3. 배포 원칙
- **점진적 배포**: dev 환경에서 먼저 검증
- **일괄 배포**: 프로덕션은 여러 기능을 모아서 한 번에 배포
- **롤백 가능성**: 문제 발생 시 이전 버전으로 즉시 복구 가능

## 11. 알려진 제약사항 및 이슈

### 현재 제약사항
- **단일 도서관 지원**: 현재 판교 도서관만 지원 (향후 전국 확장 예정)
- **청구기호 정확도**: 자동 매칭의 한계로 인한 오류 가능성
  - 해결책: 사용자 신고 시스템 + "준비 중" 플레이스홀더 처리
- **Data4Library API 속도**: 외부 API 의존으로 인한 응답 지연 가능성
  - 해결책: 캐싱 및 타임아웃 설정

### 운영 고려사항
- 신고 처리는 현재 수동 운영 (자동화 미구현)
- 대출 상태는 실시간이 아닌 5분 단위 캐시 데이터
- 스크래핑 기반 데이터 수집은 도서관 웹사이트 구조 변경 시 영향받을 수 있음
- 전국 사용자가 도서관 소장 정보를 조회할 때 "소장 없음" 케이스 증가 예상 → 교보 구매 CTA가 fallback으로 항상 활성화되어야 함
